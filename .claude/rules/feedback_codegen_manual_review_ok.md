When a function's whole purpose is to generate Pascal source code as a string for the user to review and paste in manually (e.g. `rad.utils.GenerateFluentCode`), it's acceptable for the output to occasionally need manual correction — this is not a defect to engineer away.

**Why:** Surfaced while building `GenerateFluentCode`: discovered that an active class helper in scope at a class's declaration site gets baked into that class's RTTI indistinguishably from real members (see [[project_delphi_compiler_quirks]] #6), so the generator can produce extra methods it has no way to detect or filter. When asked how to proceed, the user's answer was "şöyle bu tip yapılarda sen fonksiyonları üret parametre olarak bilgi geç bunları elle düzeltiriz. mecburen olması gerek zaten" (you generate the functions, pass info as parameters, we'll fix these by hand — it has to be that way anyway) — i.e. they already expected/accepted manual review as part of the workflow, not a gap to close.

**How to apply:** Don't chase code-level fixes for limitations that are fundamentally unfixable at the layer you're working in (e.g. runtime RTTI can't undo something baked in at compile time). Instead: (1) verify the limitation is real via an actual failing test/repro, not just theorize about it, (2) document it prominently in the function's doc comment and the corresponding docs/*.md ("üretilen kodu her zaman gözden geçir"), (3) don't add a permanently-failing test to "prove" the limitation — remove it once documented, since this project doesn't keep known-failing tests around (see [[feedback_md_autoupdate]] on doc timing generally, and [[project_delphi_compiler_quirks]] for the specific RTTI facts).
