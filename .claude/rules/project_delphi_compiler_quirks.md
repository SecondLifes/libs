Genuine dcc32 (RAD Studio 37.0) compiler/toolchain quirks discovered while building `Core\rad.eventbus.pas`, confirmed via minimal standalone `.dpr` repros (not project-code-specific bugs — will recur anywhere in this codebase/toolchain).

1. **`TQueue<TProc>` (or any `TQueue<T>` where T is a zero-arg `reference to procedure`) breaks `Dequeue`.** `dcc32` misinfers the return type as "Procedure of object" and errors E2010 "Incompatible types... Procedure of object" on any assignment of the `Dequeue` result — even in a 10-line isolated `.dpr` with no project code involved. A `TypeName(Q.Dequeue)` cast does NOT work either (E2089 Invalid typecast). Fix: don't put closures directly in `TQueue<T>`/similar generic containers — box them as `IInterface` first (the same GpEventBus-style type-erasure trick already used in this codebase: `IInterface(Pointer(@Proc)^)` to box, `TProc(BoxedIntf)` to unbox), and use `TQueue<IInterface>`.

2. **A unit cannot have a bare `finalization` section without `initialization`.** `finalization ... end.` alone (no preceding `initialization`) is E2029 "Declaration expected but 'FINALIZATION' found" — confirmed via minimal repro. Delphi requires `initialization` (even empty) before `finalization`.

3. **Any unit that `uses` an mORMot `mormot.core.*` unit fails standalone `dcc32` compilation with F2613 "Unit 'sysutils' not found"** unless `-NS` (unit scope names) is passed explicitly. mORMot source uses short/namespace-less references (`uses sysutils, classes, variants, ...`) that only resolve via Delphi's unit-scope-name search, which `.dproj`-driven builds set automatically but a bare `dcc32 file.pas` call does not. Fix: pass `-NS"System;Xml;Data;Datasnap;Web;Soap;Vcl;Vcl.Imaging;Vcl.Touch;Vcl.Samples;Vcl.Shell;Winapi;System.Win;Data.Win;Datasnap.Win;Web.Win;Soap.Win;Xml.Win;Bde"` (taken from this project's `Aksa.dproj` `DCC_Namespace` settings) alongside `-U` for the mORMot core source folder.

4. **`dcc32`'s incremental compilation can silently skip recompiling a unit that genuinely changed**, when multiple standalone `.dpr` scratch files in different directories share the same `-U` search path to a common unit (e.g., several scratch runners all pointing at `Core\rad.eventbus.pas` via `-U`). Symptom: you edit the `.pas`, recompile a `.dpr` that uses it, get a clean compile, but the NEW code path (e.g., a newly-added `{$IFDEF DEBUG}` guard) doesn't actually run — because a stale `.dcu` from an earlier compile (possibly in a different directory) was reused. Confirmed by forcing `-B` (full rebuild, ignores timestamps) which made the new code path work immediately. Fix: when a scratch verification "doesn't behave as expected" right after a source edit, rerun with `-B` before concluding the code itself is wrong.

5. **`System.Rtti`'s `TRttiStructuredType.GetDeclaredProperties` returns EMPTY for `record` properties** (Delphi 13.1 Athens / dcc32 37.0) — even for a property with the exact same shape (private `Get`/`Set` accessor methods) that reflects fine on a `class`. Confirmed the gap is real (not a visibility/filter bug) by adding the widest possible `{$RTTI EXPLICIT METHODS([vcPrivate,vcProtected,vcPublic,vcPublished]) PROPERTIES([...]) FIELDS([...])}` directive directly above the record and recompiling — no change. `GetDeclaredMethods`/`GetDeclaredFields` on records were NOT independently re-verified beyond this. Practical takeaway: don't rely on RTTI-based property reflection for `record` types in this toolchain; only `class` is safe.

6. **An ACTIVE class helper (`class helper for X`) in scope at a class's declaration site gets baked into that class's compile-time RTTI as if genuinely declared** — `TRttiMethod.Parent` for the helper's methods reports the extended class itself, not the helper, so it's indistinguishable from a real declared member at the RTTI level (`Meth.Parent <> RttiType` filtering does NOT catch it). Confirmed with `DUnitX.Utils.TObjectHelper` (`class helper for TObject`, declares `Log`/`Status`/`WriteLn`): any class declared in a unit that (transitively) `uses DUnitX.TestFramework` picks up those three methods in `GetDeclaredMethods`, even though the class never mentions them. No known runtime-RTTI fix exists for this — it must be handled by the caller (manual review of RTTI-driven output) rather than filtered in code.

7. **A closure created inside a `for` loop that is itself nested inside ANOTHER anonymous method does NOT reliably capture a fresh per-iteration copy of a loop-local `var`**, even with Delphi's "inline var" syntax (`for var idx := 0 to N do begin var X := ...; procedure(...) begin ...X... end; end`) that correctly captures fresh-per-iteration in a normal (non-nested) procedure/method (confirmed working pattern already used in `rad.thread.Tests.pas`'s `EszamanliCokGorevBasariylaTamamlanir`). Confirmed via minimal repro (`rad.thread.pas`'s original `TRadTask.WhenAll` fix attempt, 2026-07-09): a wrapper `OnFinally` closure created inside a `for` loop inside the aggregate task's own `FProc` closure (i.e., closure-inside-closure) always read the LAST loop iteration's captured value, regardless of `for-in` vs `for-to`/`var idx`, and regardless of whether the captured var itself was inline-`var`'d fresh each iteration. Isolated via a standalone debug `.dpr` that reproduced it with plain closures + `Writeln` tracing (no TRadTask involved), ruling out an app-logic bug. Fix: extract the per-iteration closure creation into a separate, NAMED (non-nested) function/method that takes the "captured" values as real parameters (e.g., `class function MakeChainedFinally(const AOriginal: TRadTaskProc; ARemaining: PInteger; ADoneEvent: TEvent): TRadTaskProc`) — a closure created inside a *regular* method capturing that method's *parameters* works reliably (same pattern already used everywhere else in this codebase, e.g. `TRadTask.FireCallback`'s `SafeInvoke` capturing `AProc`); only closure-inside-closure-inside-loop is broken.

**Why:** All seven wasted significant debugging time — the first three because the error messages/line numbers were misleading, the fourth because it looked exactly like a logic bug (silently "not triggering") rather than a stale-build artifact, #5/#6 because they looked like bugs in newly-written RTTI-reflection code (`rad.utils.GenerateFluentCode`), and #7 because it looked exactly like an app-level closure-capture mistake (the classic "loop variable capture" bug) even after applying the textbook inline-`var` fix for that classic bug — only isolating it in a closure-only, TRadTask-free repro proved it was the double-nesting, not the loop.

**How to apply:** When writing new `rad.*.pas` units in this codebase — if a class needs a `TQueue<T>` of closures, box as `IInterface` from the start rather than debugging E2010 later. If a unit needs `finalization`, always write `initialization` (blank if nothing to do) immediately before it. Any standalone `dcc32` invocation (for quick verification outside the .dproj) on a unit that pulls in `mormot.core.*` needs the `-NS` switch above. If a scratch `.dpr`'s behavior doesn't match a just-made source change, add `-B` to the compile command BEFORE spending time debugging the "wrong" behavior — it's often just a stale `.dcu`. If writing RTTI-based reflection/codegen (like `GenerateFluentCode` in `rad.utils.pas`), don't trust `record` property reflection, and expect active class helpers (esp. `DUnitX.Utils.TObjectHelper`, always in scope in test units) to pollute `GetDeclaredMethods` results in a way that cannot be filtered — document it as a caller-must-review limitation instead of chasing a code fix. If a closure needs to be created PER-ITERATION of a loop that is itself inside another closure (e.g., wiring up N callbacks from within another callback), do NOT rely on inline-`var` loop capture — extract the per-item closure factory into its own named method taking real parameters, exactly like `TRadTask.MakeChainedFinally`.
