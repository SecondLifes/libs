# rad.cmd.pas Kod Incelemesi

Kaynak: `src/core/rad.cmd.pas`  
Ilgili testler: `src/test/unit/rad.cmd.Tests.pas`  
Ilgili dokuman: `docs/rad.cmd.md`

## Genel Degerlendirme

`rad.cmd.pas`, isimle kaydedilen closure tabanli komutlari senkron/asenkron calistirmak icin sade ve kullanisli bir command registry sunuyor. `TDictionary<string, TCmd>` + `TRWLock` karari proje hafizasiyla uyumlu; `ToLowerInvariant` kullanimi da Turkce locale kaynakli "I/i" sorununu dogru kapatiyor.

Mevcut yapi temel kullanim icin iyi, fakat asenkron tarafta iki ciddi omur/yasam dongusu riski var. Bunlar kullanici tarafinda nadir ama yakalandiginda zor teshiş edilen AV veya gecersiz exception nesnesi davranisi uretir.

Bu inceleme statiktir; bu oturumda Delphi derlemesi veya DUnitX test calistirmasi yapilmadi.

## Bulgular

### 1. Yuksek: `ExecuteAsync` queued `OnError` callback'ine gecersiz exception nesnesi tasiyabilir

Konum: `src/core/rad.cmd.pas:160`, `src/core/rad.cmd.pas:161`, `src/core/rad.cmd.pas:163`

`ExecuteAsync` icinde exception `except on E: Exception do` ile yakalaniyor ve `E`, `TThread.ForceQueue` icindeki anonim metoda kapatiliyor:

```pascal
on E: Exception do
  if Assigned(OnError) then
    TThread.ForceQueue(nil, procedure begin OnError(E); end);
```

Delphi'de exception nesnesinin omru `except` blogu ile sinirlidir. `ForceQueue` callback'i daha sonra ana thread'de calisacagi icin `E` artik serbest bir nesne olabilir. Bu, `OnError(E.Message)` gibi masum bir kullanimda bile use-after-free/AV riski dogurur.

Oneri:

- `AcquireExceptionObject` ile exception nesnesinin sahipligini callback'e tasiyip callback sonunda `Free` et.
- Daha basit ama bilgi kaybettiren alternatif: `E.ClassName` ve `E.Message` degerlerini string olarak kopyala, queued callback icinde yeni bir exception uret.
- Uzun vadede `OnError: TProc<Exception>` yerine `OnError: TProc<string, string>` veya proje tipi bir error-record dusunulebilir.

Ornek yon:

```pascal
on E: Exception do
begin
  LError := Exception(AcquireExceptionObject);
  TThread.ForceQueue(nil,
    procedure
    begin
      try
        OnError(LError);
      finally
        LError.Free;
      end;
    end);
end;
```

### 2. Yuksek: `ExecuteAsync` registry omrunu garanti etmiyor

Konum: `src/core/rad.cmd.pas:150`, `src/core/rad.cmd.pas:152`, `src/core/rad.cmd.pas:157`

`ExecuteAsync`, `TTask.Run` icine `Execute(Name, Sender, Args)` cagrisini koyuyor. Bu closure instance `Self` uzerinden calisir. Cagiran kod `ICmds` referansini hemen birakirse `TCmds` nesnesi task baslamadan veya task calisirken destroy edilebilir.

Riskli kullanim:

```pascal
CreateCmds.ExecuteAsync('is', nil, [], DoneProc, ErrorProc);
```

Bu durumda `CreateCmds` ile olusan interface gecici oldugu icin `ExecuteAsync` doner donmez nesne serbest kalabilir. Task ise daha sonra `Self.Execute` cagirmaya calisir.

Oneri:

- `ExecuteAsync` icinde `Self as ICmds` seklinde bir interface referansi yakalanip task closure'ina tasinsin.
- Alternatif: `ExecuteAsync` `ITask` dondursun ve dokumanda caller'in registry omrunu garanti etmesi gerektigi acik yazilsin.

### 3. Orta: Komut adi normalize edilirken `Trim` uygulanmiyor

Konum: `src/core/rad.cmd.pas:101`, `src/core/rad.cmd.pas:111`, `src/core/rad.cmd.pas:123`, `src/core/rad.cmd.pas:172`

Komut anahtari `Name.ToLowerInvariant` ile normalize ediliyor ama bas/son bosluklar korunuyor. Bu nedenle `RegisterCommand('Kaydet', ...)` ile `Execute(' Kaydet ')` farkli komut sayilir. `rad.eventbus.pas` kanal adlarinda `Trim.ToLowerInvariant` kullaniyor; command registry icin de ayni sozlesme daha tutarli olur.

Oneri:

- Private `NormalizeName(const Name: string): string` fonksiyonu ekle.
- `Trim.ToLowerInvariant` kullan.
- Bos sonuc icin `EArgumentException` firlat.

Ek test:

```pascal
Komutlar.RegisterCommand('Kaydet', Cmd);
Assert.IsTrue(Komutlar.Exists(' kaydet '));
```

### 4. Orta: `RegisterCommand` bos ad veya nil fonksiyon kabul ediyor

Konum: `src/core/rad.cmd.pas:97`, `src/core/rad.cmd.pas:101`, `src/core/rad.cmd.pas:128`

Su an `RegisterCommand('', Func)` veya `RegisterCommand('x', nil)` engellenmiyor. Bos isim registry icin anlamsiz bir anahtar uretir. Nil `TCmd` ise `Execute` sirasinda command bulunmus gibi gorunup `LFunc(Sender, Args)` noktasinda hata uretir.

Oneri:

- `Name.Trim = ''` ise `EArgumentException`.
- `not Assigned(Func)` ise `EArgumentException`.
- Testlerde bu iki kontrat sabitlensin.

### 5. Orta: `TryExecute` butun command hatalarini "komut yok" ile ayni Boolean'a indiriyor

Konum: `src/core/rad.cmd.pas:131`, `src/core/rad.cmd.pas:136`

`TryExecute`, hem komut bulunamadiginda hem de komutun kendi icindeki exception'da `False` donuyor ve exception bilgisini kaybediyor. Dokumanda bunun tasarim tercihi oldugu yaziyor, fakat pratikte veri dogrulama, DB veya izin hatalari "komut yok" gibi algilanabilir.

Oneri:

- Mevcut API korunabilir.
- Ek olarak `TryExecute(...; out Res: TValue; out Err: Exception): Boolean` veya `out ErrorMessage: string` overload'u eklenebilir.
- En azindan `ExecuteMissing` ve `ExecuteRaised` ayrimini veren bir enum sonuc tipi dusunulebilir.

### 6. Orta: `ExecuteAsync` hata yoksa da callback exception'larini sarmalamiyor

Konum: `src/core/rad.cmd.pas:158`, `src/core/rad.cmd.pas:159`, `src/core/rad.cmd.pas:162`, `src/core/rad.cmd.pas:163`

Komutun calismasi try/except icinde, fakat `OnDone` veya `OnError` callback'inin kendisi ana thread'de exception firlatirsa bu exception sarmalanmiyor. VCL/FMX uygulamalarinda bu global exception handler'a duser; console testlerinde ise farkli davranabilir.

Oneri:

- Callback exception'lari icin ya bilincli olarak "caller sorumlulugu" diye dokumante et.
- Ya da `OnDone` callback'i de try/except ile sarilip `OnError`'a yonlendirilsin. `OnError` exception firlatirsa tekrar yutulmasi veya global logger'a aktarilmasi gerekir.

### 7. Dusuk: `Sender` ve `TValue` argumanlarinin async omru dokumante edilmeli

Konum: `src/core/rad.cmd.pas:12`, `src/core/rad.cmd.pas:25`, `src/core/rad.cmd.pas:150`

`Sender: TObject` ham pointer olarak task'a geciyor. Cagiran UI/component nesnesi async task bitmeden free edilirse command icinde gecersiz pointer kullanilabilir. Benzer sekilde `TValue` icindeki object/interface/record payload'larin thread ve omur guvenligi caller'a ait.

Oneri:

- Dokumana "async cagriya verilen Sender ve Args degerlerinin task bitene kadar gecerli kalmasi caller sorumlulugudur" notu ekle.
- Kritik yerlerde `Sender` yerine interface/weak reference/ID tabanli baglam tercih edilebilir.

## Ekleme Onerileri

### `NormalizeName` helper'i

Tek noktadan isim sozlesmesi:

```pascal
function NormalizeName(const Name: string): string;
begin
  Result := Name.Trim.ToLowerInvariant;
  if Result = '' then
    raise EArgumentException.Create('Command name cannot be empty');
end;
```

### `Clear` ve `CommandNames`

Operasyonel/diagnostic kullanim icin faydali olur:

```pascal
procedure Clear;
function CommandNames: TArray<string>;
```

`CommandNames` read lock altinda snapshot alip donmeli.

### Typed kolaylik overload'lari

`TValue` esnek ama kullanimda verbose. Siklikla kullanilacak senaryolar icin yardimci overload dusunulebilir:

```pascal
function ExecuteInt(const Name: string; const Args: TArray<TValue>): Integer;
function ExecuteString(const Name: string; const Args: TArray<TValue>): string;
```

Ya da daha genel olarak generic `ExecuteAs<T>` eklenebilir.

### Async sonuc kontrolu

`ExecuteAsync` su an fire-and-forget. Iptal, await veya test kolayligi icin:

```pascal
function ExecuteAsync(...): ITask;
```

Bu, var olan procedure overload'u bozmadan eklenebilir.

## Test Onerileri

Mevcut testler temel kayit, case-insensitive lookup, unregister, async basari/hata ve eszamanli erisimi kapsiyor. Eklenmesini onerdigim testler:

- `ExecuteAsync` icinde fırlayan exception `OnError`'a gecerli `E.Message` ile ulasiyor mu.
- `CreateCmds.ExecuteAsync(...)` gibi gecici interface kullanimi AV uretmeden tamamlanabiliyor mu.
- `RegisterCommand('', Cmd)` hata veriyor mu.
- `RegisterCommand('x', nil)` hata veriyor mu.
- `RegisterCommand(' Kaydet ', Cmd)` ve `Execute('kaydet')` sozlesmesi ne olacaksa testle sabitlenmeli.
- `TryExecute` command icindeki exception'da `False` donuyor mu ve yeni eklenecek overload hata mesajini koruyor mu.
- `OnDone` exception firlatirsa beklenen davranis neyse testlenmeli.

## Oncelikli Aksiyon Plani

1. `ExecuteAsync` icindeki exception nesnesi omru duzeltilsin.
2. `ExecuteAsync` task closure'i registry instance'ini interface referansiyla hayatta tutsun.
3. `NormalizeName` eklenip `Trim.ToLowerInvariant` tek noktadan kullanilsin.
4. `RegisterCommand` bos isim/nil fonksiyon guard'lari eklensin.
5. `TryExecute` icin hata bilgisini koruyan overload eklensin veya mevcut tasarim dokumanda daha sert vurgulansin.
6. Async `Sender/Args` omru ve callback exception davranisi dokumante edilsin.

## Kisa Sonuc

`rad.cmd.pas` mimari olarak dogru yonde ve onceki `TDynArrayHashed`/locale problemlerinden arinmis gorunuyor. En kritik duzeltme asenkron hata tasima kodunda: `E` exception nesnesi queued callback'e dogrudan kapatilmamali. Ikinci kritik konu ise `ExecuteAsync` calisirken `ICmds` instance'inin omrunu garanti etmek.
